### GeoSurgical 核心架构蓝图

GeoSurgical 的核心定位不是传统 GIS 控制台，而是一个面向空间数据的“语言驱动手术台”：用户上传数据后，用自然语言描述清洗、筛选、坐标转换、字段处理等需求，系统将意图翻译成可执行的标准指令，再由本地高性能引擎完成处理。

整体原则：

- **前端不做复杂表单堆叠**：避免把用户拖进坐标系选择、字段映射、重命名规则等配置面板。
- **JavaScript 不碰重度二进制解析**：`.zip`、`.shp`、`.dbf` 等文件扫描、解压、字段读取全部交给 Worker + Rust WASM。
- **LLM 不接触真实坐标数据**：大模型只理解用户意图和 Rust 回传的数据摘要，不读取完整几何坐标。
- **Rust WASM 不绑定固定按钮**：Rust 核心接收 JSON 指令集，并按指令动态执行空间处理任务。
- **重计算全部本地化**：真实文件解析、几何计算、坐标转换、导出都在浏览器本地 Worker 线程完成。
- **撤销能力按文件规模分级**：小文件允许保留阶段快照，大文件默认只保留操作日志和重新执行能力，避免内存被多份 500MB 副本撑爆。

#### 阶段一：基座搭建与 LUI 入口 —— 主线程只负责接待，不负责拆包

由于 GeoSurgical 是工具型应用，不依赖 SEO，也不能把用户的空间数据上传到服务端渲染，因此技术基座采用纯 SPA，而不是 Next.js SSR。

- **技术栈**：Vite + React 19 + TypeScript + Tailwind CSS
- **核心目标**：搭建文件上传 + 自然语言指令输入的轻量入口，同时确保主线程不解析大文件。

关键实现：

1. **Dropzone 文件入口**
   - 支持 `.zip`、`.shp`、`.geojson` 等常见 GIS 数据格式。
   - 主线程只负责接收文件对象，并通过 `File.arrayBuffer()` 得到原始 `ArrayBuffer`。
   - 主线程不解压 zip，不解析 DBF，不读取 Shapefile header，不扫描几何体。

2. **盲传 Worker**
   - 主线程拿到 `ArrayBuffer` 后，立刻通过 `postMessage(buffer, [buffer])` 使用 Transferable Objects 把所有权转移给 Worker。
   - 一旦 transfer 完成，主线程中的 `ArrayBuffer.byteLength` 会变成 `0`，后续不能再使用它。
   - 所有读写闭环都必须在 Worker 内完成，主线程只保留文件名、文件大小、任务 ID 等轻量 UI 状态。

3. **GUI 向 LUI 转型**
   - 去掉传统复杂清洗配置面板。
   - 不再为“坐标系选择”“字段重命名”“面积过滤”“属性清洗”分别设计一堆表单。
   - 用户通过自然语言描述任务，让系统自动翻译为可执行指令。

4. **魔法指令控制台（Command Palette）**
   - 文件上传后，页面核心交互变成一个指令输入框。
   - 示例输入：
     - “把所有面积小于 10 的块删掉，坐标转火星。”
     - “把 name 字段为空的要素过滤掉，然后导出 GeoJSON。”
     - “把地块按 area 从大到小排序，并删除重复编号。”
   - Command Palette 需要保留历史指令、执行状态、错误提示和可撤销记录。
   - 在 Rust 回传 Metadata 前，输入框可以允许用户先写需求，但不能执行 The Brain 调度。

5. **动态快捷指令（Onboarding Hooks）**
   - LUI 不能只给用户一个空输入框，否则普通业务用户会不知道系统能做什么。
   - Rust 回传 Metadata 后，前端根据字段、CRS、文件类型和 warning 自动生成 3-4 个快捷指令 Tag。
   - 示例：
     - 检测到 `crs = EPSG:4326`：生成“一键纠偏至火星坐标”。
     - 检测到存在 `area` 字段：生成“清理 area 为 0 的废弃多边形”。
     - 检测到 `name` 字段空值较多：生成“删除 name 为空的要素”。
     - 检测到缺失 `.prj`：生成“补充原始坐标系信息”。
   - 用户点击 Tag 后，系统直接把对应自然语言填入 Command Palette，再交给 The Brain 生成 AST。

#### 阶段二：Rust 前台分诊（Dry Run）—— 元数据提取也不能留在 JS

原始方案中让主线程用 JS 提取字段名、要素数量和 BBox，这在 500MB zip Shapefile 场景下会直接卡死页面。正确做法是让 Rust WASM 先兼职“前台分诊护士”：只做轻量二进制扫描，回传摘要，不展开完整数据。

- **技术栈**：Web Worker API + Rust WASM + Transferable Objects
- **核心目标**：用 Worker + Rust 完成元数据 Dry Run，让主线程永远不做重度二进制操作。

关键实现：

1. **Worker 持有原始 Buffer**
   - Worker 接收主线程盲传过来的 `ArrayBuffer`。
   - Worker 将其封装为内部任务上下文，例如 `TaskContext { buffer, metadata, status }`。
   - 后续 The Brain 生成 AST 后，主线程只需要把 AST 发给 Worker，不再重复传大文件。

2. **Rust 提供轻量元数据函数**

```ts
extract_metadata(buffer: Uint8Array): string
```

   - `extract_metadata` 返回 JSON 字符串。
   - Rust 只读取必要 header、central directory、DBF 字段描述区、Shapefile bbox 等轻量结构。
   - 对 `.zip` 场景，Rust 只定位必要条目并做最小化扫描，避免在 JS 中完整解压。
   - 目标是拿到字段名、字段类型、要素数量估计、BBox、CRS 线索、编码线索等。

3. **Metadata 回传主线程**

```json
{
  "fileType": "shapefile_zip",
  "fileName": "parcel.zip",
  "fileSize": 524288000,
  "featureCountEstimate": 12842,
  "fields": [
    { "name": "id", "type": "string" },
    { "name": "area", "type": "number" },
    { "name": "name", "type": "string" }
  ],
  "bbox": [113.91, 22.45, 114.37, 22.86],
  "crs": "EPSG:4326",
  "encoding": "GBK",
  "fieldPolicy": {
    "totalFieldCount": 200,
    "includedFieldCount": 50,
    "truncated": true
  },
  "warnings": []
}
```

4. **Metadata 截断与采样**
   - Metadata 不是越全越好，尤其本地小模型上下文有限。
   - 字段数量超过 50 个时，Rust 只回传优先级最高的前 50 个字段，并标记 `truncated: true`。
   - 字段优先级可按以下规则排序：几何/面积/长度相关字段、ID 字段、名称字段、非空率高的字段、用户历史常用字段。
   - Rust 可以为每个字段附带最多 3 个轻量 sample，用于帮助 LLM 理解字段语义。
   - sample 必须脱敏、截断，并限制总大小；不能把整列属性表塞进 Prompt。
   - 示例字段：

```json
{
  "name": "land_type",
  "type": "string",
  "sample": ["住宅用地", "商业用地", "工业用地"],
  "nullRateEstimate": 0.03
}
```

5. **分诊失败策略**
   - 如果文件损坏、缺少 `.dbf`、投影信息缺失或编码不确定，Rust 返回结构化 warning。
   - 主线程把 warning 显示在 Command Palette 附近，让用户可以继续用自然语言补充约束。
   - 例如：“投影文件缺失，请输入原始坐标系。”用户输入后再交给 The Brain 生成 AST。

#### 阶段三：大模型调度中枢（The Brain）—— Metadata 到 AST

只有在 Rust Dry Run 回传 Metadata 之后，系统才唤醒大模型语义网关。The Brain 的职责是把用户自然语言和 Rust 元数据翻译为标准化 GeoSurgical AST。

- **输入**：用户自然语言 + Rust 回传 Metadata
- **输出**：标准 JSON 指令集（GeoSurgical AST）
- **边界**：LLM 绝不接触完整坐标、完整属性表或原始文件内容。

调度流程：

1. **Prompt 构造**
   - 主线程将用户输入和 Metadata 打包发送给 LLM。
   - LLM 可以来自云端 API，也可以来自本地 Ollama。
   - Prompt 核心约束：只输出 JSON，不解释，不生成代码，不请求原始坐标。
   - Prompt 必须遵守 Metadata 的截断边界：如果 `truncated: true`，禁止让 LLM 假装看过未提供字段。
   - 当用户提到未出现在 Metadata 中的字段时，The Brain 应返回“需要字段检索”类 AST，而不是猜测字段名。

2. **GeoSurgical AST 输出协议**
   - LLM 输出一组可审计、可校验、可回放的 JSON 指令。
   - 示例：

```json
{
  "version": "1.0",
  "operations": [
    {
      "action": "filter_area",
      "operator": ">=",
      "field": "area",
      "value": 10
    },
    {
      "action": "transform_crs",
      "from": "EPSG:4326",
      "to": "GCJ-02"
    },
    {
      "action": "export",
      "format": "geojson"
    }
  ]
}
```

3. **指令安全校验**
   - 前端在发送给 Worker 前先校验 AST。
   - 只允许白名单 `action`。
   - 校验字段是否存在、参数类型是否正确、数值范围是否合理。
   - 对高风险操作展示确认，例如大规模删除、坐标系不可逆转换、覆盖导出等。

4. **二次下发**
   - 主线程只把 AST 和任务 ID 发给 Worker。
   - Worker 复用阶段二已经持有的原始 `ArrayBuffer`。
   - 不允许主线程重新读取、复制或解析原始文件。

5. **字段检索补偿**
   - 如果用户自然语言提到的字段没有出现在前 50 个 Metadata 字段里，主线程向 Worker 发起轻量字段检索请求。
   - Worker 调用 Rust 在完整字段目录中查找候选字段，只回传匹配字段摘要，不回传完整属性表。
   - The Brain 拿到补充字段摘要后再生成正式 AST。

#### 阶段四：异步计算通道 —— Worker 管住文件生命周期

Worker 是浏览器端的计算边界：它拥有文件内存、调用 Rust WASM、接收 AST、回传进度和结果。

- **技术栈**：Web Worker API + Transferable Objects
- **核心目标**：把文件解析、WASM 调用、进度回传与主线程 UI 隔离。

关键实现：

1. React 主线程只负责 UI 状态、文件选择、Command Palette、AST 预览和结果展示。
2. Worker 在任务上下文中保存原始 `ArrayBuffer`，并负责 Dry Run 与正式执行两个阶段。
3. Worker 内部加载 Rust WASM 模块：
   - 分诊阶段调用 `extract_metadata(buffer)`。
   - 执行阶段调用 `execute_surgery(task_id, json_instructions)` 或 `execute_surgery(buffer, json_instructions)`。
4. Worker 持续回传进度、日志、警告、错误和最终结果。
5. 用户可以在主线程取消任务，Worker 接收取消信号后中断后续操作。

#### 阶段五：Rust WASM 动态指令引擎 —— 从功能执行器升级为手术引擎

Rust WASM 不再暴露一堆硬编码函数，例如 `convert_wgs84(buffer)`、`filter_area(buffer)`。它至少暴露两个稳定入口：

```ts
extract_metadata(buffer: Uint8Array): string
execute_surgery(buffer: Uint8Array, json_instructions: string): Uint8Array
```

更推荐的 Worker 封装方式是让 Worker 持有 buffer，Rust 正式执行时只接收任务上下文和 AST，避免 JS 层重复搬运大对象。

Rust 的定位是“JSON 指令解析与执行引擎”，负责解析 GeoSurgical AST，并通过 Dispatcher 动态分发到底层空间处理方法。

- **技术栈**：Rust + `wasm-bindgen` + `serde` + georust 生态（如 `shapefile-rs`、`geo-types`、`geo`）
- **核心目标**：让新增能力优先通过扩展 AST 协议和 LLM 提示词完成，而不是每次新增前端按钮和 Rust 对外函数。

关键实现：

1. **元数据 Dry Run**
   - Rust 只扫描必要头部信息和目录结构。
   - 不展开全部几何体，不构建完整要素集合。
   - 输出轻量 JSON Metadata，供 The Brain 使用。

2. **统一执行入口**
   - JS 正式执行只触发 `execute_surgery`。
   - Rust 内部负责解析文件、解析指令、执行操作、返回结果。

3. **GeoSurgical AST 解析**
   - 使用 `serde` 定义强类型指令结构。
   - 对未知 `action`、缺失字段、类型错误、非法参数直接返回结构化错误。

4. **Dispatcher 分发器**
   - 根据 JSON 指令数组顺序执行操作。
   - 示例映射：
     - `filter_area` -> 面积过滤模块
     - `transform_crs` -> 坐标转换模块
     - `rename_field` -> 字段重命名模块
     - `drop_empty` -> 空值过滤模块
     - `export` -> 格式导出模块

5. **进度条心跳**
   - Rust 处理大批量要素时必须定期向 Worker 汇报进度。
   - 建议每处理 1000 条要素或每完成一个 operation 阶段发送一次 progress event。
   - Worker 将进度转发给主线程，Command Palette 可以实时显示：
     - “正在扫描字段：完成”
     - “正在过滤面积：45%”
     - “正在转换坐标：78%”
     - “正在导出 GeoJSON：92%”

6. **撤销与快照策略**
   - Rust 内部必须明确每个 operation 是原地修改还是生成新版本。
   - 对不可逆操作（删除要素、字段丢弃、坐标纠偏）不能依赖“逆向计算”来撤销。
   - 推荐分级策略：
     - 小文件（例如 50MB 以下）：保留上一个阶段的快照指针或轻量版本快照，允许一步撤销。
     - 中等文件：只保留关键 checkpoint，例如执行前原始 buffer + 当前 AST 日志，撤销时重新执行到上一步。
     - 大文件（例如 500MB）：默认禁用完整撤销，只允许取消当前任务、重新从原始文件执行新 AST。
   - Command Palette 中的“撤销”必须基于 Worker 返回的能力标记决定是否可用，不能前端硬开按钮。
   - Worker 应回传：

```json
{
  "undo": {
    "available": false,
    "reason": "file_too_large",
    "strategy": "replay_from_original"
  }
}
```

7. **动态流水线**
   - 每条操作接收上一步的中间结果。
   - 操作链可组合、可回放、可记录。
   - 未来新增能力时，优先新增一个 Rust 内部 operation handler，并教 LLM 输出对应 JSON。

8. **基础能力**
   - 解析 Shapefile / DBF / GeoJSON。
   - 处理 GBK、UTF-8 等常见属性编码问题。
   - 执行字段清洗、属性过滤、几何过滤、坐标转换、格式导出。
   - 输出 GeoJSON 或压缩后的 Shapefile。

#### 阶段六：可视化验证 —— OpenLayers 渲染与结果审阅

处理结果回到主线程后，使用 OpenLayers 展示空间数据，让用户在导出前进行可视化核验。

- **技术栈**：OpenLayers / WebGL
- **核心目标**：让用户看到处理前后差异，而不是只下载一个黑盒结果。

关键实现：

1. React 接收 Worker 返回的 GeoJSON、Blob URL 或结果文件摘要。
2. OpenLayers 使用 `VectorSource` 加载处理后的 GeoJSON。
3. 大数据量场景优先使用 WebGL 渲染能力，例如 `WebGLVectorLayer`。
4. 提供处理前后对比、要素数量变化、字段变化、坐标范围变化等摘要信息。
5. 用户确认无误后再导出最终文件。

#### 推荐执行链路

完整链路如下：

1. 用户拖拽上传 GIS 文件。
2. 主线程获取 `ArrayBuffer`，不做任何解析。
3. 主线程通过 Transferable Objects 将 `ArrayBuffer` 盲传给 Worker。
4. Worker 持有原始 buffer，并调用 Rust `extract_metadata` 做 Dry Run。
5. Rust 只扫描必要二进制头部和目录结构，回传 Metadata JSON。
6. Rust 对字段进行截断与采样，避免 Metadata 撑爆 LLM 上下文。
7. 主线程展示 Metadata 摘要、warning 和动态快捷指令 Tag。
8. 用户点击快捷 Tag 或在 Command Palette 中输入自然语言需求。
9. The Brain 将“自然语言 + Metadata”转换为 GeoSurgical AST。
10. 如字段缺失，主线程向 Worker 发起轻量字段检索，再让 The Brain 补全 AST。
11. 前端校验 AST，并展示可读化执行计划。
12. 用户确认后，主线程只把 AST 发给 Worker。
13. Worker 复用已持有的 buffer，调用 Rust WASM 的 `execute_surgery`。
14. Rust Dispatcher 按 AST 动态执行空间处理流水线，并定期上报进度。
15. Worker 返回处理结果、日志、摘要和撤销能力标记。
16. OpenLayers 渲染结果，用户审阅后导出。

#### 必须遵守的工程约束

- **主线程不拆包**：任何 zip 解压、DBF 解析、Shapefile header 读取都不能放在 React 主线程。
- **Buffer 所有权单行道**：`ArrayBuffer` transfer 给 Worker 后，主线程不能再读写它。
- **Worker 持有任务上下文**：同一个任务的 Dry Run 和正式执行必须复用 Worker 内的 buffer。
- **LLM 只看摘要**：The Brain 只能接收 Metadata、用户指令和 AST schema，不能接收原始坐标。
- **Metadata 必须控量**：字段过多时必须截断，字段 sample 必须限制数量、长度和总 token 预算。
- **空白输入框不可裸奔**：Metadata 回传后必须生成动态快捷指令，降低用户冷启动成本。
- **进度必须有心跳**：Rust Dispatcher 处理长任务时必须定期发 progress event，避免 LUI 看起来像卡死。
- **撤销能力必须分级**：根据文件大小和操作可逆性决定快照、回放或禁用撤销。
- **错误必须结构化**：Rust 和 Worker 返回的错误要包含 `code`、`message`、`recoverable` 和可选 `suggestedUserInput`。

#### 架构收益

- **交互更轻**：用户不用学习复杂 GIS 表单，直接描述目标。
- **冷启动更低**：系统根据 Metadata 自动生成快捷指令，让用户知道下一步能做什么。
- **主线程更稳**：大文件二进制解析不进入 React 主线程，LUI 不会被 500MB zip 卡死。
- **上下文更稳**：Metadata 截断与采样避免 LLM 被 200 个字段和海量枚举值拖垮。
- **隐私更强**：LLM 只看元数据，不看真实坐标。
- **扩展更快**：新增能力主要扩展 AST 与 Dispatcher，而不是重画 UI。
- **性能更稳**：重计算放到 Worker + Rust WASM，不阻塞 React 主线程。
- **结果可控**：AST 可审计、可校验、可回放，避免大模型直接操作数据。
